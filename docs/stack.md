# ADR-0001 — Tech stack and repository tooling

**Status:** Accepted (v0.0.5) · **Epic:** E01 · **Module:** MOD-01
**Context source:** the project's architecture reference and dev plan
(maintained by the core team separately).

## Decision

kantaq is a **hybrid Python-core + TypeScript-UI** application:

| Concern | Choice | Why |
|---|---|---|
| Core (protocol, sync, MCP gateway, domain, CLI) | **Python 3.12**, **FastAPI** runtime | One language for the security-critical core; FastAPI gives OpenAPI for free. |
| MCP server | official **Python MCP SDK**, HTTP transport (stdio in v0.3) | First-party SDK; loopback HTTP is the lowest common denominator. |
| Local database | **SQLite (WAL)** with **SQLModel** + **Alembic** | Zero-setup local replica; one model definition, two dialects. |
| Crypto | **PyNaCl** (Ed25519), golden vectors checked in | Battle-tested libsodium binding. |
| Web UI | **React + Vite** (TypeScript), built static, served by FastAPI | One process, one port (3939), one token. |
| Layer contract | **OpenAPI** generated by FastAPI; the TS client is regenerated in CI | No hand-written client drift. |
| Env & packages | **uv** (+venv) for Python, **pnpm** for `web/` | Fast, reproducible, lockfile-based. |
| Tests & lint | **pytest, ruff, mypy** (Python); **Vitest, Biome** (web) | Fast, single-binary linters. |

The runtime serves the built UI as static files, so a member runs **one process on
one port** (default `3939`) with one token.

## Repository layout

See the [README](../README.md#repository-layout). A `uv` workspace rooted at the
repo top; each member under `packages/*` and `apps/*` is its own package with a
`src/` layout. The umbrella `kantaq` package carries the version and the `kantaq`
CLI, and depends on every member so a single `uv sync` installs the whole workspace.

## Golden-rule reuse decisions (E01 tooling)

Per the Golden rule (reuse before build), the bootstrap tooling below was chosen
over building anything from scratch. All clear the license bar (no GPL/AGPL) and
the maintenance/credibility bar.

| Need | Chosen | License | Notes |
|---|---|---|---|
| Python env + workspace | **uv** (Astral) | Apache-2.0 / MIT | Fast resolver, native workspaces, built-in lockfile. |
| Python lint + format | **ruff** (Astral) | MIT | Replaces flake8/isort/black; single binary. |
| Python type checker | **mypy** | MIT | Strict mode on the core. |
| Python tests | **pytest** | MIT | De-facto standard. |
| Web runtime API | **FastAPI** + **Starlette** + **uvicorn** | MIT / BSD | OpenAPI-native; async; widely used. |
| Web build | **Vite** + **React** | MIT | Fast dev server + static build. |
| Web tests | **Vitest** | MIT | Vite-native test runner. |
| Web lint + format | **Biome** | MIT / Apache-2.0 | Single fast binary for lint + format. |
| Web routing (E18) | **react-router-dom** | MIT | SPA routing for the 5-page shell; React Router team, ~54k stars. |
| Runtime config (E22) | **pydantic-settings** | MIT | `.env` loading + validation; pydantic org; pairs with FastAPI/pydantic already in the stack. |
| Property tests (E27) | **hypothesis** | MPL-2.0 | Property-based tests for the harness + sync profiles; HypothesisWorks, ~7.5k stars. MPL-2.0 is reuse-compatible. |
| Test fakes (E27) | **built from scratch** (stdlib) | Apache-2.0 (ours) | `FakeClock`/`SeededRandom`/`FakeBackend` are tiny + injectable. freezegun (<5k stars) globally monkeypatches time — heavier and less explicit than an injected clock. |

| Conventional-commit lint | **conventional-pre-commit** | MIT | commit-msg hook. |

When a feature module starts, run its own Golden-rule pass for the libraries that
module needs (PyNaCl, SQLModel, Alembic, the MCP SDK, the OpenAPI→TS generator,
etc.) and record the choice here.

### E02 / MOD-02 data layer

| Need | Chosen | License | Notes |
|---|---|---|---|
| ORM + schema-as-code | **SQLModel** | MIT | Pydantic + SQLAlchemy in one model; FastAPI author; ~16k stars; shares Pydantic with the rest of the stack. Beat **raw SQLAlchemy** (more boilerplate, no Pydantic bridge) and **Tortoise/Peewee** (own migration story, weaker typing). Architecturally mandated by ADR-0001. |
| Migrations | **Alembic** | MIT | The canonical migration tool for SQLAlchemy/SQLModel, by the SQLAlchemy maintainers — the only option that diffs against `SQLModel.metadata` (`compare_metadata`). Below the 5k-star bar in isolation, but it ships as part of the SQLAlchemy ecosystem (credible org, no advisory) and has no real alternative for this stack; **yoyo-migrations** and **migra** don't read SQLAlchemy metadata. Mandated by ADR-0001. |
| ULID ids | **built from scratch** (`kantaq_db.ids`, stdlib) | Apache-2.0 (ours) | Architecture §6 requires ULID ids. No ULID library clears the 5k-star bar (`python-ulid` ~0.6k, `ulid-py` ~0.5k), so we implement the spec directly: 48-bit ms timestamp + 80-bit randomness, Crockford Base32, monotonic factory. ~90 lines + tests. We implement the real spec (sortable, timestamp-recoverable) rather than claim a standard we don't honor — the test harness made the same call when it renamed its look-alike to `sortable_id`. |
| Postgres driver (tests) | **psycopg** (v3) | LGPL-2.1 (driver, dynamically linked) | Dev/test-only, for the live SQLite/Postgres parity check; not shipped in the runtime path. Standard Postgres driver, psycopg org. |

### E06 / MOD-06 identity & grants

| Need | Chosen | License | Notes |
|---|---|---|---|
| Token hashing (Argon2id) | **cryptography** (pyca) | Apache-2.0 / BSD | ~7k stars, PyCA org, very active, no unpatched advisory. | Token entropy + lookup | **secrets** (stdlib) + keyed format | PSF / ours | `kq_<token_ulid>.<secret>` — the row id travels in the token (GitHub/Stripe pattern) so verification is one lookup + one Argon2id check, not O(active tokens). |
| OS keychain | **built from scratch** (`FileKeychain`, 0600 file) | Apache-2.0 (ours) | No Python keychain library clears the bar: **keyring** (~1.3k stars), **secretstorage** (Linux-only, <1k), **pyobjc keychain** (macOS-only). v0.0.5 parks the runtime token in a 0600 file beside the SQLite replica; the real OS-keychain backend is re-evaluated with the v0.1 device keys (D-01), which actually need it. ### E24 / MOD-05 Supabase backend

| Need | Chosen | License | Notes |
|---|---|---|---|
| Supabase API client (Auth) | **built from scratch on httpx** (`kantaq_backend_supabase.auth`, ~150 lines) | Apache-2.0 (ours) | The official Python stack all fails the 5k-star bar: **supabase-py** ~2.5k★, **supabase/auth-py** ~0.1k★, **supabase/postgrest-py** ~0.3k★ (verified via the GitHub API, 2026-06). v0.0.5 needs five GoTrue REST calls (otp / verify / refresh-token / user / logout, all documented), so a thin typed client over **httpx** (15k★, BSD-3, already in the stack) beats adopting a below-bar SDK plus its transitive pins. Revisit supabase-py at v0.2 (atomic RPC + Realtime). |
| JWT claim parsing (key guard) | **PyJWT** | MIT | ~5.7k★, active, the focused RFC 7519 implementation. Beat **python-jose** (~1.8k★, below bar, history of advisories) and **Authlib** (~5.3k★ — clears the bar but is a full OAuth/OIDC framework; we only read claims). Used unverified-decode only: the client has no signing secret by design, the guard is a local misconfiguration check (refuse a `service_role` key), and Supabase verifies keys server-side. |
| RLS test infrastructure | **EphemeralPostgres** (MOD-30, kept) + a Supabase auth-env stub + `TamperedClient` (built, stdlib + SQLAlchemy) | Apache-2.0 (ours) | Resolves the MOD-30 open question (testcontainers vs disposable local Postgres): **testcontainers-python** is ~2.2k★ (below bar) and adds a Docker-API dependency; **pytest-postgresql** is below bar. The existing EphemeralPostgres (disposable database on a provided server, CI Postgres service) already proved itself in E02; E24 adds a faithful stub of Supabase's documented auth environment (`anon`/`authenticated`/`service_role` roles, `auth.jwt()`/`auth.uid()` reading `request.jwt.claims`) so the *checked-in* policies are tested against the contract they run under in production. |
| Authorization model | **PostgreSQL Row Level Security** (policies in `supabase/policies/`) | n/a (SQL standard feature) | The real, server-enforced standard — no homegrown authz DSL. Policies follow Supabase's published conventions (SECURITY DEFINER helper functions against RLS recursion; per-verb policies; `service_role` with BYPASSRLS staying backend-side). |
| Sync endpoints (E24-T4, Sprint 2) | **PostgREST's documented REST dialect** (Supabase's data API), thin client on **httpx** (`kantaq_backend_supabase.sync`, ~200 lines) | PostgREST MIT (server, run by Supabase); ours Apache-2.0 | Golden-rule re-run 2026-06 (GitHub API): **supabase-py** 2,518★ and **supabase/postgrest-py** 317★ stay below the bar; **PostgREST** itself (27,220★, MIT, active) *is* the recognized standard here — Supabase runs it, so v0.0.5 needs zero server code. We implement its documented dialect properly: bulk insert with `on_conflict` + `Prefer: resolution=ignore-duplicates` (= Postgres `ON CONFLICT DO NOTHING`, the idempotent-re-push contract), `return=representation` for commit-order revisions from the identity column, `gt./eq.` filters + `order`/`limit` paging. Realtime stays out (polling is the v0.0.5 baseline; revisit with v0.2's RPC). Tested against the dialect for real: MOD-30's `FakePostgREST` answers the same requests with real SQL + RLS on a disposable Postgres. |

**Verification cost vs the 5 s revocation budget (NFR-E06-2).** Argon2id is
deliberately slow (~50 ms), too slow for every request, so the runtime caches
verified tokens in memory for **3 s** (< 5 s): a revoked token keeps working at
most TTL seconds across processes, and same-process revocation purges the cache
immediately. The cache stores a SHA-256 digest of the presented token, never the
token itself.

**D-07 (single-source dialect generation) — resolved as "shared SQLModel."** The
8 collections are defined once as SQLModel models; both SQLite and Postgres DDL
are generated from that one `MetaData` via SQLAlchemy's dialect compilers (no
hand-kept second schema). Parity is proven two ways: an offline structural
fingerprint that asserts both dialects describe the same schema (hermetic, runs
everywhere) and a live `EphemeralPostgres` reflection diff in CI. To keep the
dialects in lock-step we use only portable column types — `str`/VARCHAR for
enum-like fields (no native Postgres `ENUM`) and generic `JSON` for list/dict
fields. The fallback in the sprint risk note (two hand-kept schemas + a CI diff)
was not needed.

### E07 / MOD-07 audit

| Need | Chosen | License | Notes |
|---|---|---|---|

| Need | Chosen | License | Notes |
|---|---|---|---|

## Consequences

- Two toolchains in CI (Python + web). Keep total CI **under 10 minutes**
  (RISK-05) via dependency caching and parallel jobs.
- FastAPI emits OpenAPI; CI regenerates the TS client and fails on drift (D-08,
  lands with E18).
- One schema definition generates both SQLite and Postgres DDL (D-07, lands with
  E02).
